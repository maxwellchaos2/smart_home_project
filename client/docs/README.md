# Client Service Documentation

## 1. Назначение
`client` — Flutter-приложение системы Smart Home. Сервис отвечает за пользовательский интерфейс, авторизацию пользователя, отображение состояния устройств, запуск сценариев, получение событий и отправку команд в серверную часть.

## 2. Зона ответственности
- Отрисовка UI для всех основных доменных сущностей (комнаты, устройства, сценарии, события).
- Валидация пользовательского ввода перед отправкой команд.
- Работа с токенами/сессией и обновлением авторизационного состояния.
- Получение телеметрии/событий и реактивное обновление интерфейса.
- Локальное кэширование данных, необходимых для быстрого старта и оффлайн-устойчивости.

- `docs/MODULES_AND_FEATURES.md` — карта пакетов `client/packages` и их роль.

## 4. Основные пользовательские сценарии
1. Пользователь открывает приложение и проходит авторизацию.
2. Приложение загружает профиль, комнаты, устройства и их статусы.
3. Пользователь отправляет команду устройству (вкл/выкл/режим/параметр).
4. Клиент отправляет запрос на backend и показывает актуальный результат.
5. Пользователь запускает сценарий или редактирует пользовательские настройки.

## 5. Быстрый обзор архитектурного старта
1. Точка входа `lib/main.dart` читает `ENV` из `--dart-define`.
2. Выбирается `DevAppConfig` или `ProdAppConfig` (`lib/src/app_config.dart`).
3. `AppRunner` инициализирует DI-контейнер (`GetIt`) и модули.
4. Через `Dio` настраивается API-клиент с `baseUrl` из app config.
5. `AppBuilder` собирает `MaterialApp`, scope-провайдеры и маршруты.
6. Навигация начинается с `SplashScreen`, затем переход в рабочие экраны.

## 6. Внешние зависимости
- Backend API (`server`) для получения/изменения состояния доменных сущностей.
- Сервис событий/стриминга для near-real-time обновления UI.
- Платформенные подсистемы Android/iOS/macOS для разрешений, push и сетевого стека.

## 7. Нефункциональные требования
- Устойчивость к кратковременной потере сети.
- Быстрое время холодного старта и плавная работа интерфейса.
- Безопасное хранение чувствительных данных на клиенте.
- Корректная деградация UX при недоступности backend.

## 8. Границы ответственности
Клиент не исполняет бизнес-логику устройств напрямую, не управляет физическим оборудованием и не хранит первичные системные данные как источник истины. Источник истины — серверные сервисы.
