# Sync Service Documentation

## 1. Назначение
`sync_service` — центральный локальный сервис Smart Hub на малинке.

Практически это «ядро» локального контура:
- другие сервисы в том же Docker Compose обращаются к нему по внутренней сети;
- он синхронизирует локальные события с удаленным сервером;
- только он хранит и использует токен авторизации устройства;
- первичная настройка (Wi‑Fi + токен + project/device id) приходит из Flutter-приложения через Bluetooth.

## 2. Ответственность
- Проксирование HTTP API в удаленный backend (`REMOTE_SERVER_URL`).
- Проксирование/локальная обработка WebSocket трафика (`/ws`).
- Инъекция `Authorization: Bearer <token>` в исходящие запросы к удаленному серверу.
- Подмена `projectId` в маршрутах `/api/projects/-1/*` на реальный `projectId` устройства.
- BLE provisioning: принять от приложения Wi‑Fi и авторизационные данные.
- Поддержание heartbeat устройства в удаленном контуре.

## 4. Переменные окружения
- `REMOTE_SERVER_URL` — удаленный backend для проксирования API/WS.
- `CONFIG_DATA_PATH` — путь к каталогу с `config.json` (по умолчанию `/app/config_data`).
- `PORTT` — порт HTTP сервиса (по умолчанию `8080`).

## 5. Данные provisioning
После сопряжения через Bluetooth сервис получает и сохраняет:
- Wi‑Fi данные (`ssid`, `pass`) для подключения малинки к сети;
- `authToken`, `deviceId`, `projectId` для авторизованной синхронизации.

Файл хранения: `config_data/config.json`.

## 6. Результат для системы
Все локальные сервисы в общем compose работают через единый sync-слой и не держат собственный auth токен к удаленному API.
